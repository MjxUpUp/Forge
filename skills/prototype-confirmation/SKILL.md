# 原型确认门（动手前先让用户拍板）

把方案/范式做成**可逐功能点确认的原型**，让用户对每个功能点的形态明确表态（认可/调整/推翻）后，再动手写真实代码。

**核心原则：自决范式、写完代码再让用户看 = 偷懒 + 返工。原型成本 << 返工成本。**

本 skill 是一道**门控**：范式级重构/交互重设计，代码落地前必须过这扇门。过了（用户逐项确认）才进 implementation-discipline 的实施链。

## When to Use / When NOT to Use

**Use**：
- 起底重构、重新设计前端交互、范式级改动（不止换壳，布局/交互范式要变）
- 多功能点方案（≥3 个独立功能点），任一形态选错都返工
- 用户说"出原型 / 让我看 / 让我确认 / 先看效果 / 原型给我看 / 确认形态"
- 自己心里已"定好范式"准备直接写代码（**STOP，这是最高危触发**）
- 大重构分块实施前（每块都要有原型对应）

**SKIP**：
- 单功能点小改、bug 修复、纯样式微调 → 直接改
- 纯后端逻辑、无交互形态 → 直接写
- 纯调研 → research-workflow
- 只改一行配置/版本号 → 直接改

## 流程（gate，每步过门才进下一个）

### 步骤 1 — 枚举功能点（基于实际代码，不凭记忆）

把方案拆成**可独立确认的功能点清单**。每个功能点：
- 有编号（A1/B2/...）+ 简短标题
- 标**来源**：对应哪块代码/哪个需求/哪个 commit（`grep`/`Read` 实证，不凭记忆枚举）
- 知道**状态变体**：空态/正常/告警/失败各长什么样

**门控**：功能点清单的每个"来源"必须是实际 grep/读出来的，不是记忆里的。凭记忆枚举 → 漏功能点或编造不存在的功能。

**红线**：凭记忆列功能点 / 不标代码来源 / 把"我以为有的功能"列进去。

### 步骤 2 — 出 HTML 原型（单文件，逐功能点展示）

单 HTML 文件（自包含 CSS/JS，不依赖构建），结构：
- **左导航**：按组（轴线/模块）列全部功能点，点击切换
- **右内容**：当前功能点的原型态
  - 编号 + 标题 + 来源标签（块/commit/代码位置）
  - "这是什么"一句话说明
  - **视觉原型**：用真实组件样式渲染 mock 数据（不是空壳）
  - **状态变体并排**：空/正常/告警/失败，让用户看到边界情况

**门控**：原型必须用项目真实设计 token（颜色/字体/圆角/间距），从 `variables.css`/设计系统镜像，不另起一套。视觉漂移会让用户确认的是一个不像成品的原型。

**红线**：原型只做总览不逐功能点 / 用空壳 mock（无真实数据）/ 另起设计 token / 只展示正常态不展示边界态。

### 步骤 3 — 原型必须内建「导出 + 持久化」（硬要求，最高频踩坑）

原型是让用户**在自己浏览器里**逐项确认的。用户勾完的状态只存在浏览器内存——agent 读不到。所以原型**必须**自带：

1. **逐项确认 UI**：每个功能点三个单选（✅认可 / ✏调整 / ❌推翻）+ 备注输入框
2. **进度条 + 导航打勾**：已确认项可视化
3. **localStorage 持久化**：勾选状态写 localStorage，刷新不丢
4. **导出按钮**：点一下把全部确认结果（含备注）汇总成文本复制到剪贴板，用户粘贴给 agent

**没有导出 + 持久化 = 确认数据回收不了**（用户确认了，agent 拿不到，等于没确认）。这是本 skill 最重要的硬要求。

**门控**：交付原型给用户前，自检原型里有导出按钮 + localStorage 写入逻辑。缺任一 → 补上再交付。

**红线**：原型只有勾选 UI 没导出按钮 / 没持久化（刷新丢）/ 假设"用户会截图告诉我"。

参考实现：本 skill 自带模板 `references/feature-tour-template.html`（自包含骨架，含导出按钮 + localStorage 持久化 + 三态勾选 + 主题切换；复制后替换 `GROUPS`/`DEMOS` 为你项目的功能点 + `:root` 镜像你项目的 `variables.css` 即可，无需从零写导出/持久化逻辑）。

### 步骤 4 — 用户逐项确认 → 导出全量

把原型路径给用户，让其逐项勾选 + 填备注。完成后点「导出确认」按钮，把剪贴板内容贴回会话。

**门控**：验收只认用户导出的全量文本（用户点导出按钮 → 复制剪贴板 → 粘贴回会话，再据用户确认实施），不靠偷读 localStorage（Gotcha 3 测试数据污染）。**核对**：用 `grep -cE '✅|✏|❌'` 数导出文本里标了判定的功能点条数，须 `==` 清单总数；缺项 = 漏勾，回找用户确认，不替用户默认"未勾=认可"。

### 步骤 5 — 据确认结果实施

- ✅认可 → 保留对应代码/设计
- ✏调整 → 按备注改（备注是用户的真实意图，逐条回应）
- ❌推翻 → 该功能点重来
- 未勾 → 找用户确认是"默认认可"还是"漏勾"，不替用户默认

**门控**：调整项里若有产品级问题（如"要不要加看板"），先给有依据的判断再实施，不只记录。

## Common Rationalizations（堵借口）

| 借口 | 现实 |
|---|---|
| "范式我心里清楚了，直接写代码更快" | 写完用户说"不是我要的"→ 整块返工。原型 1 小时，返工 1 天起 |
| "原型做总览就够了，功能点不用逐个展开" | 总览看不出边界态（空/失败/告警）。用户确认的是模糊印象，落地才发现细节不对 |
| "用户会截图告诉我确认结果" | 截图漏备注、不可检索、易截不全。导出按钮一键复制全量，没理由用截图 |
| "原型先不做导出，回头补" | 用户确认完才发现没导出 = 数据全丢（刷新/关页就没）。导出必须原型第一版就带 |
| "功能点我记得，列一下就行" | 凭记忆列会漏功能点/编造功能。grep 实际代码枚举，每个标来源 |
| "mock 数据随便填" | 空壳 mock 让用户确认的是一个无信息量的壳。用真实场景数据，用户才能判断形态对不对 |
| "测试原型时我点的勾，应该不影响用户" | 测试数据会混进用户确认（尤其共享 storage 时）。测试完清掉脏数据，或隔离实例 |

## Red Flags — STOP（看到这些想法，你在 rationalize）

- "这个范式我定好了，直接写代码" → STOP，先出原型
- "原型做个总览让用户看个意思" → STOP，逐功能点 + 状态变体
- "导出按钮回头再加" → STOP，导出+持久化是第一版硬要求
- "功能点我记得有哪些" → STOP，grep 实际代码
- "原型用默认样式就行，不用对齐设计 token" → STOP，视觉漂移让确认失真
- "用户截图给我就行" → STOP，要导出按钮的文本
- "测试时点的勾没事" → STOP，清脏数据或隔离实例

## Gotchas（真实失败，最高信号）

- **6 块代码全写完才出原型**：agent 自决范式，写完 6 块 + 砍 CLI 才让用户看。用户当场质问"为什么没有出原型让我确认？"。→ 范式级改动，第一块代码落地前就要出原型。自决范式 + best-judgment proceed 是越权。
- **原型没导出 + 没持久化，确认数据回收不了**：原型只有勾选 UI，没导出按钮、没 localStorage。用户"已经逐项确认好"后 agent 读不到，只能让用户补导出。→ 导出按钮 + 持久化是原型第一版硬要求，不是 v2 补丁。
- **测试实例与用户浏览器共享 localStorage，测试数据污染用户确认**：agent 测试原型时自己点了几个勾（带编的测试备注），与用户浏览器共享 storage，导出时测试备注混进用户确认，差点把测试值当用户反馈实施。→ 测试原型后清掉脏数据（或用独立 profile）；验收只认用户导出的全量文本，不认 agent 偷读的 localStorage；备注语气/内容若像测试值要显式向用户核对。
- **凭记忆枚举功能点，漏/错**：凭印象列功能点清单，漏掉已落地的功能或编造不存在的。→ grep/Read 实际代码，每个功能点标代码位置（文件:行）。
- **原型只展示正常态**：每个功能点只 mock 一个正常场景，用户确认后落地才发现空态/失败态/告警态长得不对。→ 每个功能点并排展示状态变体。

## 与其他 skill 的分工

| 范畴 | 本 skill | 指向谁 |
|---|---|---|
| 动手前范式形态确认 | 本 skill 专管（gate） | — |
| 方案要有依据（技术选型/API） | 阶段 0 衔接 | evidence-based-proposal |
| 代码实施全程纪律 | 确认后才进 | implementation-discipline |
| 项目交付后验收 | 不做（本 skill 是动手前） | project-acceptance |
| 纯调研 | 不做 | research-workflow |

**与 implementation-discipline 的衔接**：implementation-discipline 阶段 0「方案要有依据」之前，若是范式级/交互重设计，先过本 skill 的原型确认门。即：原型确认（本 skill）→ 方案依据（evidence-based-proposal）→ 实施纪律链（implementation-discipline 阶段 0-5）。
