# 架构决策记录

记录重要的技术决策，包含上下文、备选方案和理由。ADR 创建可搜索的知识库，避免重复讨论已定论的决策，帮助新成员（或未来的 AI 会话）快速了解背景。

## 何时创建 ADR

**创建**：在 2+ 方案中选择时、采用或替换工具时、定义团队应遵循的模式时。
**不创建**：显而易见的选择、临时变通方案、风格偏好。

## 阶段 0 — 决策发现（Inversion）

**在决策被清晰框定之前，不要开始撰写。**

向用户提问：

1. "需要做什么决策？用问题形式表述。"
2. "目前考虑了哪些选项？"
3. "有什么约束条件？（性能、兼容性、团队经验、时间线）"
4. "如果决策失误，影响有多大？（可逆 vs 不可逆）"

收集答案后呈现：**"决策问题是：[问题]。选项是：[A, B, C]。关键约束：[列表]。是否开始研究和评估这些选项？"**

**用户确认框定后才继续。**

## 阶段 1 — 研究与撰写（Generator）

```
ADR 创建：
- [ ] 1. 研究备选方案
- [ ] 2. 评估权衡
- [ ] 3. 撰写 ADR
- [ ] 4. 提交审批
- [ ] 5. 建立索引
```

### 步骤 1：研究备选方案

对每个选项：擅长什么、不足什么、与现有代码的集成度、社区状态。

### 步骤 2：评估权衡

使用决策矩阵：

| 标准 | 选项 A | 选项 B | 选项 C |
|------|--------|--------|--------|
| 性能 | ++ | + | +++ |
| 复杂度 | + | ++ | - |
| 维护性 | ++ | ++ | + |

### 步骤 3：撰写 ADR

加载并填写 [templates/adr-template.md](templates/adr-template.md)。

存储位置：`docs/adr/NNNN-title.md`（零填充序号）。

### 步骤 4：提交审批

**门控：呈现完整 ADR。用户批准或要求修改前不要提交。**

### 步骤 5：建立索引

追加到 `docs/adr/README.md`。

## ADR 状态

- **Proposed（提议中）** → **Accepted（已接受）** → **Deprecated（已弃用）** / **Superseded by ADR-NNNN（被取代）**

## 易错点

- **不要补写所有历史决策**：只记录新决策。
- **保持简洁**：最多 1-2 页。
- **链接到代码**：引用实现该决策的文件、模块、PR。
- **记录被否决的方案**："未选择的方案"部分往往最有价值——防止重复提议。
- **日期很重要**：上下文随时间变化；2025 年的决策在 2027 年可能不适用。
