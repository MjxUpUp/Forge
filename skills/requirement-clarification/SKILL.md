# 需求澄清（模糊→可验收规格）

把用户脑子里、口头一句话、或分散在多段对话里的模糊需求，澄清为**清晰、可执行、可验收**的规格。Forge 研发全流程 gate-1（需求定义）的原子能力——上游产出可直接喂给 `doc-generator` 生成 PRD，或作为 `implementation-discipline` 的输入。

## 核心原则

**先问后做**。模糊需求直接动手 = 用 30 分钟写错方向。花 3 分钟串行提问，把方向锁死再动手。

> **Iron Law**：DO NOT start designing / coding / writing the PRD until the requirement passes the clarity gate in §决策树. 一句话需求几乎一定藏着至少 3 个未说出口的假设。

---

## When to Use / When NOT to Use

### Use when（任一命中即触发，宁可多触发）

- 用户只给一句话："帮我做个 dashboard" / "加个搜索功能" / "搞个 CLI 工具"
- 需求描述含模糊量词："快一点"、"用户体验好"、"支持高并发"、"轻量级"、"够用就行"
- 验收条件不可观测、不可测试（"看起来对"、"差不多就行"、"专业感"）
- 范围边界不清：不知道什么在 Scope、什么在 Out of Scope
- 隐性假设未暴露：技术栈、用户画像、运行环境、数据规模、性能基线
- 需求评审前，需要把口头需求转成可签字的规格
- "做完了"的标准无法回答——问"怎么算完成"，答不上来
- 跨人协作交接：A 给的需求转给 B 实现，中间必须落规格

### SKIP（明确排除，提供 skill 间互斥）

| 场景 | 用什么 | 为什么 |
|---|---|---|
| 需求变量已全部明确，只缺 PRD 文档结构 | `doc-generator` | doc-generator 管"按模板填空"，本 skill 管"澄清未知变量" |
| 要做技术方案选型 / 第三方库论证 / 可行性 | `evidence-based-proposal` | 前者管"怎么做"，本 skill 管"做什么" |
| 需求已清晰，直接进入编码循环 | `implementation-discipline` | 那里管"实现纪律"，本 skill 已完成使命 |
| 要进入 Plan Mode 整体规划多任务 | `dev-workflow` | dev-workflow 是规划编排器，本 skill 是它**规划前**的澄清原子能力 |

## Inversion Gate（硬 gate）

输出任何方案、PRD、代码之前，先跑 §澄清维度 的串行提问。**用户对全部维度给出明确回答，且通过 §决策树 的"清晰可执行"判定后，才解锁后续动作。**

```
用户给一句话需求
   ↓
[Inversion Gate] 串行提问（§澄清维度，一次一组）
   ↓
通过 §决策树 的"清晰可执行"判定？
   ├─ 否 → 继续问下一组，或显式标记 assumption 写进 Out of Scope
   └─ 是 → 输出 §输出规格模板，可 HANDOFF 给 doc-generator / implementation-discipline
```

---

## 澄清维度（七维，缺一不可问）

按顺序串行澄清，**不要一次性堆七组问题**——用户会跳过、会乱答、会疲劳。

### 1. 范围边界（Scope）
- 要解决的具体问题是什么？（一句话，不是一段话）
- 谁是用户 / 谁是使用者 / 谁是维护者？（三种角色不一定相同）
- 触发场景：什么情况下用？什么时候不用？

### 2. 验收条件（Acceptance Criteria，必须可观测可测试）
- 完成后怎么验证？（不是"看起来对"，要"运行 X 命令看到 Y 输出"或"用户能完成 Z 操作且耗时 < N 秒"）
- 最小可用（MVP）包括什么？什么算"够用"？
- 每条验收条件必须能转成测试用例或可重复的演示步骤

### 3. Out of Scope（明确不做的）
- 哪些相邻功能**不在**本次范围？（写下来比写 Scope 更重要——防范围蔓延）
- 后续可能扩展但不做的事有哪些？

### 4. 约束（Constraints）
- 技术：必须用 / 不能用什么栈？已有系统的兼容性约束？
- 资源：人力、预算、外部依赖（API、账号、license）
- 时间：硬 deadline？还是宽松探索？
- 运行环境：OS、网络、权限、隔离要求

### 5. 优先级（Priority）
- 多个目标冲突时，什么优先？（性能 vs 功能 vs 工期 vs 可维护性）
- 必须有（P0）vs 应该有（P1）vs 可以有（P2）

### 6. 模糊术语消歧（Disambiguation）
- 用户说的"快"、"轻量"、"专业"、"好用"、"现代"——具体数值或参考物是什么？
- 行业黑话 / 缩写——双方理解一致吗？

### 7. 隐性假设暴露（Hidden Assumptions）
- 你（agent）准备开工前，**默认成立**的所有前提是什么？写出来让用户确认。
- 数据规模、并发量、输入格式、错误处理策略、国际化、安全模型——任一未明示都是潜在地雷。

## 决策树：什么算"足够清晰可执行"

对每个维度逐项过，**任一回答"否"或"不确定"，回到 §澄清维度 继续问**：

```
该维度是否已澄清？
   ↓
├─ 范围：能否用一句话说清要解决的具体问题？
│   ├─ 否 → 问"具体解决谁的什么问题，什么场景下"
│   └─ 是 ↓
├─ 验收：完成后的状态能否转成可重复的测试 / 演示步骤？
│   ├─ 否 → 问"做完后我怎么验证给你看？跑什么命令、看到什么结果"
│   └─ 是 ↓
├─ Out of Scope：是否有≥3条明确"不做"的事项？
│   ├─ 否 → 问"哪些相邻功能你不打算这次做？"
│   └─ 是 ↓
├─ 约束：技术栈 / 资源 / 时间 / 运行环境四项是否都明确？
│   ├─ 否 → 问缺失项
│   └─ 是 ↓
├─ 优先级：冲突时哪个优先是否明确？
│   ├─ 否 → 问"如果工期和功能冲突，砍哪个？"
│   └─ 是 ↓
├─ 术语：所有模糊量词是否已替换为数值或参考物？
│   ├─ 否 → 问"快 = 多少 ms / 多少 QPS？轻量 = 多少 MB / 多少依赖？"
│   └─ 是 ↓
└─ 假设：我准备开工前的默认前提，用户是否全部确认？
    ├─ 否 → 列出你的假设清单让用户 yes/no
    └─ 是 → 全部通过，进入 §输出规格模板
```

**任一维度实在问不出答案时**：不要替用户猜。明确标记为 **ASSUMPTION** 写进 Out of Scope 或 Constraints 段，由用户后续修正——比"我猜你大概是这个意思"破坏性小得多。

---

## 串行提问协议

**一次只问一组**，避免问题轰炸：

```
错误：把七个维度的问题一次性抛给用户（10 个问题）
       → 用户跳过一半、乱答一半、疲劳放弃

正确：
  第 1 轮：问维度 1（范围）+ 简短为什么需要这信息
  用户答完 →
  第 2 轮：问维度 2（验收）+ 复述第 1 轮结论（"确认我理解对..."）
  用户答完 →
  ... 直到七维全部澄清
```

每轮包含三段：
1. **复述上一轮结论**（确认理解一致，防止累积偏差）
2. **本轮问题**（一组，不超过 3 个相关问题）
3. **为什么需要这信息**（让用户知道回答的价值，对抗"为什么要问这么多"）

**短路条件**：用户在某一轮说"你自己定" / "按行业惯例" / "随便"——立即把对应维度标记为 ASSUMPTION，写进规格，**不要反复追问**——那是在制造摩擦。

## 输出规格模板

七维全部通过后，输出以下结构（这是 HANDOFF 给 `doc-generator` / `implementation-discipline` 的产物）：

```markdown
# 需求规格 — <一句话标题>

## 问题陈述
<一句话：解决谁的什么问题，什么场景下>

## Scope（在范围内）
- ...
- ...

## Out of Scope（明确不做）
- ...
- ...

## 验收条件（可观测可测试）
- [ ] AC1: <可重复的验证步骤 + 期望输出>
- [ ] AC2: ...
- [ ] AC3: ...

## 约束
- 技术：<栈 / 兼容性 / 已有系统>
- 资源：<人力 / 预算 / 外部依赖>
- 时间：<deadline 或探索性>
- 运行环境：<OS / 网络 / 权限>

## 优先级
- P0 必须：<...>
- P1 应该：<...>
- P2 可以：<...>

## 术语消歧
- "<模糊词>" = <数值 / 参考物 / 明确定义>

## 假设（需用户后续确认或修正）
- ASSUMPTION-1: <我假设 X，因为未明示>
- ASSUMPTION-2: ...

## 后续
- 直接 HANDOFF 给 doc-generator 生成 PRD
- 或作为 implementation-discipline 的输入开始编码
```

## Common Rationalizations（堵借口的最高信号）

| 借口 | 现实 |
|---|---|
| "用户说做个 dashboard，我直接做就行" | "dashboard" 含义 100 种——是运维监控？业务数据？用户行为？不问清楚就是 30 分钟写错方向 |
| "需求看起来挺清楚的" | 看起来清楚 ≠ 验收条件可测试。问"怎么算完成"答不上来 = 没清楚 |
| "这些细节问太多用户会烦" | 一次问一组、说清为什么，比做完返工更省用户时间 |
| "技术栈我猜一下，行业都这么做" | 通用知识 ≠ 当前项目环境。这是 evidence-based-proposal 的红线，本 skill 也适用 |
| "Out of Scope 用户没说，我先做出来再说" | 没写 Out of Scope = 范围蔓延保证书。明确"不做"比"做"更防争议 |
| "假设我替用户定了就行" | 替用户定的假设会被当成"你承诺的"。写 ASSUMPTION 让用户认领，不是替他决策 |
| "用户说随便，那我就按最佳实践" | "随便" = 用户授权你决定。但要写进规格让他后续能修正，不是悄悄固化 |
| "验收条件我自己心里有数" | 心里有数 ≠ 写下来。不写 = 跨人交接必然丢 |

## Red Flags — STOP 先问再做

如果你正要做这些，STOP：

- 收到一句话需求，立刻打开编辑器 / PRD 模板 / 代码文件
- 用模糊量词描述目标："做得快一点"、"体验更好"、"支持高并发"
- 默认了技术栈、运行环境、用户画像但没和用户确认
- Out of Scope 段是空的或只有一句"无"
- 验收条件用"看起来对"、"差不多"等不可测试词
- 跳过了某个澄清维度因为"这个不重要"
- 用户说"随便" / "你定"后，把决定默默做下去而不写进规格

**以上都意味着：你在猜不在澄清。回到 §澄清维度 继续问，或显式写 ASSUMPTION 让用户认领。**

## Gotchas（来自实际失败的需求陷阱）

### Gotcha 1: 隐性假设的爆炸半径
- **问题**：用户说"做个搜索功能"，agent 默认了"全文检索 + 中文分词 + Elasticsearch"
- **现象**：做完用户说"我只要 Ctrl+F 那种"
- **解决**：每个未明示的技术决策都是潜在假设。维度 7 强制列出"我准备开工前的默认前提"，让用户 yes/no 一遍

### Gotcha 2: 验收条件不可测
- **问题**：验收写成"系统运行流畅"
- **现象**：上线后用户说"还是卡"，agent 说"我这边测试很流畅"
- **解决**：每条验收条件必须能转成可重复的命令或步骤——具体输入 → 期望输出。模糊主观词必须替换为数值（P99 延迟、QPS、错误率）

### Gotcha 3: 范围蔓延（Scope Creep）
- **问题**：Out of Scope 段空着，开发过程中用户不断加"顺便也做一个..."
- **现象**：工期翻倍，每个新加的都"看起来不大"，累积爆炸
- **解决**：Out of Scope 必须至少列 3 条。每条用"本次不做，理由 X"格式。后续加项必须显式变更 Scope

### Gotcha 4: 未说出口的约束
- **问题**：用户没提"必须离线运行"、"必须支持 IE11"、"数据不能出内网"
- **现象**：做完发现部署环境根本没外网 / 浏览器不兼容 / 合规不允许
- **解决**：维度 4 约束项**主动问**运行环境、网络、权限、合规——用户不会主动说，因为对他来说是"默认的"

### Gotcha 5: 优先级冲突时的隐性取舍
- **问题**：用户说"既要快、又要全、还要省"
- **现象**：工期紧张时 agent 自己砍了功能，用户暴怒
- **解决**：维度 5 优先级**预先**问冲突取舍——"如果工期和功能冲突砍哪个"。冲突发生时按预先约定执行

### Gotcha 6: 术语双方理解不一致
- **问题**：用户说"轻量级"，agent 理解为"代码量少"，用户实际指"依赖少、二进制小"
- **现象**：做完了发现方向错——重构成本高
- **解决**：维度 6 强制把模糊词替换为数值或参考物——"轻量"="二进制 <5MB 且无外部非静态依赖"

### Gotcha 7: 跨人交接的需求衰减
- **问题**：A 给 agent 澄清完，结果交接给 B 实现，B 没看到规格
- **现象**：B 按自己理解做，和 A 的预期偏差
- **解决**：输出规格必须**落盘成文件**（HANDOFF.md 或 PRD 草稿），不能只留在对话里。后续 implementation-discipline 从盘上读规格，不从对话历史读

## 执行后自查清单（产出规格前的最后一道门）

输出 §输出规格模板 前，逐项对照：

- [ ] 范围：能用一句话说清要解决的具体问题？（不是一段话）
- [ ] Scope：每条都是"做什么"，可执行？
- [ ] Out of Scope：至少 3 条明确"不做"？
- [ ] 验收条件：每条都能转成可重复的测试或演示步骤？（含具体输入和期望输出）
- [ ] 验收条件：没有"看起来"、"差不多"、"流畅"、"专业"等不可测词？
- [ ] 约束：技术 / 资源 / 时间 / 运行环境四项都填了？
- [ ] 优先级：冲突取舍预先约定了？
- [ ] 术语：所有模糊量词已替换为数值或参考物？
- [ ] 假设：所有未明示但默认成立的前提，已显式写为 ASSUMPTION-<N>？
- [ ] 后续：明确指向 doc-generator（生成 PRD）还是 implementation-discipline（直接编码）？

**任一答"否"，回到对应维度继续澄清，不要放出未通过的规格。**

## 与其他 skill 的分工

| 下游 skill | 本 skill 输出怎么用 | 边界 |
|---|---|---|
| `doc-generator` | 把 §输出规格模板 的内容填进 PRD 模板 | doc-generator 管"按模板填空 + 文档结构"，本 skill 管"填什么内容" |
| `evidence-based-proposal` | 本 skill 锁定"做什么"后，evidence-based-proposal 论证"怎么做" | 本 skill 不出技术方案，只出需求规格 |
| `implementation-discipline` | 本 skill 输出的验收条件 = 实现完成标准 | 实现纪律管"怎么不偏题"，验收条件来自本 skill |
| `dev-workflow` | 本 skill 是 dev-workflow 规划前的需求澄清原子能力 | dev-workflow 是规划编排器（多任务规划），本 skill 是单任务的需求澄清 |

**关键纪律**：本 skill 的输出（§输出规格模板）必须落盘成文件后再 HANDOFF。口头或对话内的规格在跨会话 / 跨人 / 跨工具交接时必然衰减。落盘后由下游 skill 从盘上读，不从对话历史读。
