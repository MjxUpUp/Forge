# 严重性分级标准

三档分级 + 评分细则。目标是让"能不能提交"的判断有客观依据，而非凭感觉。

---

## 分级定义

### 🚫 Block（必须修复才能提交）

**判定标准**：满足任一即 `block`：

1. **AI 作弊模式**（见 ai-cheat-patterns.md 任一条命中）
   - 断言弱化、错误吞没、假修复、假重构、覆盖率侵蚀、测试松绑、类型抑制、幻觉 mock、注释充数、异常上下文丢失、死分支

2. **正确性 bug**
   - 逻辑错误会导致错误结果
   - 边界条件未处理会导致崩溃（空指针/越界/除零）
   - 数据丢失或损坏风险
   - 竞态条件

3. **安全漏洞**
   - 硬编码密钥/凭证
   - SQL 注入/命令注入/XSS/路径穿越
   - 认证/授权缺失或可绕过
   - 缺输入验证导致可注入

4. **编译/运行时必然失败**
   - 假重构导致 ReferenceError
   - 类型错误被 ignore 压制但运行会崩
   - 缺失依赖/幻觉包

5. **资源泄漏**
   - 文件/连接/锁未释放
   - 无界增长（channel/缓存/列表）

**处理**：明确说"🚫 不可提交"，列出所有 block 项及修复方向。修复后需重新审查。

---

### ⚠️ Fix（应当修复，建议提交前处理）

**判定标准**：满足任一即 `fix`（非必须，但不修会增加维护成本）：

1. **可维护性问题**
   - SRP 违反（函数/类过大，多职责）
   - DRY 违反（复制粘贴的逻辑）
   - 过度设计（YAGNI 违反，无用抽象）
   - 抽象层级混乱

2. **设计味道**
   - 深继承、上帝类、特性依恋、散弹式修改
   - 不合适的设计模式（滥用或缺失）

3. **错误处理缺陷（非吞没）**
   - 错误消息不友好/缺调试信息
   - 失败路径无清理
   - 缺少必要日志

4. **测试缺陷**
   - 覆盖率侵蚀（新分支无测试）
   - 测试名不描述行为
   - 只测 happy path 不测错误路径
   - mock 不真实（但模块存在，只是 mock 简化）

5. **性能问题**
   - N+1 查询
   - 循环内 I/O
   - 不必要的克隆
   - 热路径的 O(n²)

6. **类型质量**
   - `any`/`Object`/`interface{}` 滥用（非 ignore 压制，是直接用弱类型）
   - 跨层类型不一致

**处理**：说"⚠️ 可提交，但建议先修复 N 项"。列出 fix 项。允许用户权衡后提交，但要明确成本。

---

### 💡 Suggest（建议考虑，不影响提交）

**判定标准**：

1. **可读性**
   - 命名可改进
   - 嵌套可减少
   - 注释可补充或精简

2. **微优化**
   - 非热路径的小优化
   - API 设计的小改进

3. **风格**
   - 不影响正确性的风格偏好
   - 代码组织的小调整

4. **未来改进**
   - 当前 YAGNI，但可记录为技术债
   - 可考虑但非必须的模式

**处理**：说"💡 可考虑"。简短列出，不强求。

## 评分细则（可选，用于量化质量）

对三个维度各打 1-10 分，附一句话理由：

### 设计质量（1-10）
- **9-10**：职责清晰，抽象得当，符合 SOLID，无明显设计味道
- **7-8**：基本合理，有 1-2 处可改进但不影响理解
- **5-6**：有设计味道（SRP/DRY 违反），但可维护
- **3-4**：多处违反 SOLID，难维护，技术债明显
- **1-2**：上帝类/面条代码/严重耦合，几乎无法维护

### 可维护性（1-10）
- **9-10**：改动局部化，有测试保护，命名清晰，错误处理完善
- **7-8**：可维护，部分需改善
- **5-6**：改动会牵连多处，测试覆盖不足
- **3-4**：改动高风险，测试形同虚设
- **1-2**：任何改动都可能破坏，无测试或测试无效

### AI 屎山风险（1-10，越低越好）
- **1-2**（低风险）：无作弊模式，测试真实有效，类型严格
- **3-4**：有 1-2 处 fix 级质量问题，但无作弊
- **5-6**：有 suggest 级的松绑迹象，或弱类型较多
- **7-8**：有 fix 级的测试/类型问题
- **9-10**（高风险）：命中任一 AI 作弊模式 → 直接此档 + block

## 决策矩阵

| 发现类型 | 数量 | 默认级别 | 提交决策 |
|---|---|---|---|
| AI 作弊 | ≥1 | block | 🚫 不可提交 |
| 安全漏洞 | ≥1 | block | 🚫 不可提交 |
| 正确性 bug | ≥1 | block | 🚫 不可提交 |
| 可维护性 fix | 1-2 | fix | ⚠️ 可提交，建议修 |
| 可维护性 fix | ≥5 | fix（考虑升级） | ⚠️ 强烈建议先修 |
| 测试缺陷 | 1-2 | fix | ⚠️ 可提交，建议补测试 |
| suggest | 任意 | suggest | ✅ 可提交 |

**特殊情况**：
- 单个 fix 但涉及核心模块（认证/支付/数据迁移）→ 升级为 block
- 多个 suggest 集中在同一文件 → 提示整体重构（但级别不变）
- fix 项超过 10 个 → 整体代码质量堪忧，建议返工而非逐条修

---

## 禁止的分级反模式

- ❌ **降级讨好**：发现 block 但说"建议修一下" → 必须明确不可提交
- ❌ **模糊结论**："基本可以""问题不大""看着改改" → 必须给明确提交/不提交判断
- ❌ **只列问题不给修复方向**：每个发现都要有"修复方向"
- ❌ **堆砌 suggest 凑数**：suggest 应是真实改进，不是为数量而列
- ❌ **忽视累积效应**：5 个 fix 各自看不大，合起来就是屎山
- ❌ **用评分代替门控**：评分是参考，门控决策以 block 存在与否为准
