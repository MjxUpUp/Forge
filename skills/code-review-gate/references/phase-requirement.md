# 环节审查：需求设计（requirement）

针对需求设计阶段产物（PRD / 需求文档 / user story）的审查清单。与 code-review-gate 的代码级审查互补不替换——**只审显式设计产物文件**，代码 diff 走 code-review-gate。

**规范来源**：IEEE 830 · 用户故事 INVEST · Google PRD template
**核心维度**：完整 / 可测 / 无歧义

---

## 1. 结构完整性

- [ ] PRD 必含 **Out of Scope** 节（明确不包含什么，降低范围蔓延风险）
- [ ] 每条需求有**验收条件**（Acceptance Criteria），非纯描述
- [ ] 验收条件**可执行/可验证**（非"用户友好"、"性能良好"等模糊表述）
- [ ] 需求间**无冲突**（两条需求对同一行为的期望一致）
- [ ] 覆盖**异常流**（失败路径、边界条件、错误处理），不只 happy path
- [ ] 有**用户角色 / 使用场景**定义（谁在什么情况下用什么功能）

## 2. 无歧义性

- [ ] 无**未量化的模糊词**（"快速"、"大量"、"适当"——应改为具体指标如"≤200ms"、"10万条"、"与现有约定一致"）
- [ ] 术语一致（同一概念全文用同一名词，不混用"订单"/"工单"/"单"）
- [ ] 优先级明确（Must/Should/Could 或 P0/P1/P2，非纯罗列）

## 3. 可追溯性

- [ ] 每条需求可追溯到**验收条件**（需求 → AC 一对一或一对多映射）
- [ ] 非功能性需求有**量化指标**（性能、安全、可用性有数字而非定性描述）
- [ ] 依赖项明确（外部依赖、上游接口、下游系统的已知约束）

## 4. 边界与假设

- [ ] 已知假设已列出（"假设用户已登录"、"假设数据已清洗"）
- [ ] 数据模型有基本定义（核心实体、关键字段、关系）
- [ ] 与现有系统的集成点已标识

---

## 确定性规则（机械可检）

| 规则 | 检测方式 | 来源 |
|------|----------|------|
| PRD 无 Out of Scope 节 | `file_not_contains` 关键词 "Out of Scope" | IEEE 830 |
| 模糊词存在 | 正则扫描 "快速\|良好\|适当\|合理\|大量\|少量" 后跟单位缺失 | 经验积累 |
| 需求无验收条件 | 扫描需求列表后无 "Given/When/Then" 或 "验收条件" 标记 | 用户故事标准 |

## 与大厂规范的映射（方向，非条文）

- **IEEE 830** → 完整性检查（Out of Scope、异常流）
- **INVEST** → 用户故事质量（独立的、可协商的、有价值的、可估算的、小的、可测试的）
- **Google PRD template** → 结构化（背景、目标、范围、指标、风险）
