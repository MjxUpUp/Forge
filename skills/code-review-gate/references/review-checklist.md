# 传统软件工程审查清单

按维度组织的完整审查清单。基于 mgreiler/code-review-checklist（1060⭐）+ Clean Code + SOLID + 业界共识。

**使用方式**：① 先 `Grep` 改动符号（改名/删除/改签名的函数·类型·常量）的全仓调用方 + 数据流下游，确认波及面都已更新——纯 diff 行审查会漏跨文件影响；② 再对每个被审文件的变更逐维度检查。不是每条都适用——按语言和场景判断。

---

## 1. 正确性 & 逻辑

- [ ] 逻辑覆盖所有分支（if/else/match/switch 穷举）
- [ ] 边界条件已处理：空输入、空集合、零、负数、最大值、off-by-one
- [ ] 数值运算考虑溢出/精度（浮点比较用 epsilon）
- [ ] 字符串处理：对外部输入的编码转换不崩溃
- [ ] 集合访问前已验证索引/键存在
- [ ] 空值/null/undefined/None/Option 在每条路径上处理
- [ ] 能想到让代码不按预期工作的输入或外部事件吗
- [ ] 并发场景：竞态、死锁、可见性（共享可变状态）

## 2. 错误处理 & 边界

- [ ] 无对重要 Result/Error 的静默忽略（`let _ =`、空 catch）
- [ ] 错误正确传播（不是 catch 后吞掉或只 log 后继续）
- [ ] 错误消息用户友好且含足够调试信息（保留 cause）
- [ ] 外部输入（用户输入、API 响应、文件内容）在信任前已验证
- [ ] 失败路径有清理（资源释放、事务回滚、状态恢复）
- [ ] 日志记录关键失败点，便于排查
- [ ] 幂等性：重试不会产生副作用（API/支付/发消息）

## 3. 可维护性（SOLID + Clean Code）

### SRP（单一职责）
- [ ] 每个类/模块/函数只做一件事
- [ ] 文件不超过 ~300-500 行（语言相关，超了考虑拆分）
- [ ] 函数不超过 ~30-50 行（超了通常职责过载）
- [ ] 函数参数不超过 ~4-5 个（超了用对象/Builder/struct）

### DRY（不重复）
- [ ] 无复制粘贴的逻辑块（相似代码应抽取）
- [ ] 相同的 magic number/字符串已提取为常量
- [ ] 相似功能在代码库中是否已存在可复用

### YAGNI（不过度设计）
- [ ] 无"将来可能用到"的抽象/接口/配置项（当前不需要就别加）
- [ ] 无未使用的代码（dead code）
- [ ] 无被过度参数化但只有一个调用的函数

### 抽象层级
- [ ] 代码在正确的抽象层级（不要在高层函数里写底层细节）
- [ ] 依赖方向一致（高层不依赖低层细节的实现）
- [ ] 模块边界清晰，无概念泄漏（实现细节不该暴露到接口）

### 依赖管理
- [ ] 无循环依赖（模块间）
- [ ] 新增依赖是必要的吗（避免依赖膨胀）
- [ ] 依赖是可信的、真实存在的（防 Slopsquatting：AI 幻觉包）
- [ ] 依赖版本固定/兼容（lockfile 已更新）

### 过度工程（delete-list）

上面 SRP/DRY/YAGNI 是散点 checkbox。过度工程还可执行化为 **delete-list**——重造标准库轮子（`stdlib:`）、引依赖做几行事（`native:`）、单实现抽象层（`yagni:`）、投机灵活性（`yagni:`）、死代码（`delete:`）、可压缩循环（`shrink:`）。AI 高频过度构建。每条给"删什么 + 替代什么"，目标是让 diff 变短。完整清单 + 懒惰阶梯根因诊断见 [over-engineering-checklist.md](over-engineering-checklist.md)。

## 4. 设计模式

- [ ] **滥用检测**：单例/工厂/抽象层在不需要时就是负担（命中 → 用 [over-engineering-checklist.md](over-engineering-checklist.md) 的 delete-list 给删改方案）
- [ ] **缺失检测**：明显的策略/状态/观察者场景是否用了合适模式
- [ ] 模式应用是否符合意图（不是为用而用）
- [ ] 继承 vs 组合：优先组合，继承层级不超过 2-3 层
- [ ] 接口隔离：接口不被迫实现不需要的方法
- [ ] 开闭原则：扩展开放，修改封闭（新增功能不需要改现有代码）

## 5. 安全（AI 代码高频问题，默认必查）

基于 Arcanum-Sec/sec-context（150+ 源）的 AI 代码 Top 10 反模式：

- [ ] **无硬编码密钥**：API key/密码/token/连接串（AI 高频，源数据排名 #23）
- [ ] **无 SQL 字符串拼接**：用参数化查询（`?` / `$1` / prepared statement）
- [ ] **无命令注入**：`exec`/`system`/`spawn` 不拼字符串，用参数数组
- [ ] **输入验证**：所有外部输入验证类型/长度/范围/格式
- [ ] **输出转义**：渲染到 HTML/SQL/Shell 前转义（防 XSS，AI 失败率 86%）
- [ ] **认证**：密码用 bcrypt/argon2/scrypt，不用 MD5/SHA（AI 误信率 75.8%）
- [ ] **授权**：每个端点检查权限，不只检查登录
- [ ] **限流**：敏感操作（登录/注册/支付）有限流
- [ ] **文件上传**：验证类型/大小/路径（防穿越和 RCE）
- [ ] **最小数据暴露**：API 返回 DTO 而非完整 ORM 对象
- [ ] **CSRF/CORS**：跨域配置正确，不 `*`
- [ ] **密钥存储**：用环境变量/密钥管理服务，不提交到仓库
- [ ] **路径穿越**：用户输入拼路径前消毒（`../`）

### DB/SQL 破坏性操作（注入之外还要查）

上面查的是 SQL **注入**（字符串拼接导致越权读写）。还要查**破坏性** DDL/DML——语法合法但摧毁数据/权限/结构：DROP DATABASE/TABLE/SCHEMA、TRUNCATE、无 WHERE 的 DELETE/UPDATE、GRANT ALL/TO PUBLIC、生产直连、不可逆迁移、未事务化多语句。AI 生成迁移/数据脚本时高频，一次失误清空生产库。完整清单 + 检测方法见 [sql-safety-checklist.md](sql-safety-checklist.md)。

## 6. 性能

- [ ] 无循环内的 I/O（DB/HTTP/文件应批量）
- [ ] 无 N+1 查询（循环里查 DB）
- [ ] 无不必要的大对象克隆/拷贝（Rust 用引用，JS 注意对象 spread）
- [ ] 热路径最小化分配（预分配容量）
- [ ] 无 O(n²) 算法（存在 O(n log n)/O(n) 替代时）
- [ ] 缓存使用合理（缓存确认的瓶颈，不是猜测的）
- [ ] 懒加载/分页用于大数据集
- [ ] 无阻塞调用在异步上下文（同步 I/O 在 async 函数里）

## 7. 测试有效性（不只看覆盖率）

- [ ] 测试名描述被测行为，不是方法名（`should_reject_expired_token` 而非 `test1`）
- [ ] 断言验证真实行为，不是 `toBeTruthy`/`toBeDefined` 凑数
- [ ] 边界情况已测：空输入、零、负数、最大值、错误路径
- [ ] 测试相互独立（无共享可变状态、无执行顺序依赖）
- [ ] 测试覆盖错误路径，不只 happy path
- [ ] 无被注释/跳过的测试（`it.skip`/`#[ignore]`）无理由
- [ ] mock 的模块真实存在（防幻觉 mock）
- [ ] 集成测试覆盖模块间交互（单元测试全绿 ≠ 系统能跑）
- [ ] 无遗留的 `console.log`/`println!` 调试输出
- [ ] 测试是可重复的（不依赖时间/网络/随机未 seed）

## 8. 可读性

- [ ] 命名清晰表达意图（`getUserById` 而非 `getData`，`isExpired` 而非 `flag`）
- [ ] 无误导性命名（`temp` 实际是最终结果、`list` 实际是 map）
- [ ] 无单字母变量（除循环索引 `i`/数学公式）
- [ ] 无注释掉的代码（git 里有历史）
- [ ] 注释解释"为什么"不是"做什么"（代码已说做什么）
- [ ] 复杂逻辑有注释说明意图
- [ ] 嵌套不超过 3 层（超了用 early return/guard clause）
- [ ] 控制流直观（避免深层嵌套的 if/else）
- [ ] 文件/模块放在正确的位置（按项目约定）
- [ ] 一致的代码风格（命名、格式、错误处理模式）

## 9. API 设计（如变更涉及对外接口）

- [ ] 公开 API 有文档（JSDoc/docstring/OpenAPI）
- [ ] 参数顺序/命名直觉
- [ ] 错误类型描述性强（不只抛 `Error`）
- [ ] 多参数用对象/struct（避免位置参数地狱）
- [ ] 向后兼容：现有调用者不会因变更而坏
- [ ] 版本化：破坏性变更有版本/迁移路径

## 10. 项目约定（按项目调整）

- [ ] ID 类型与项目约定一致（UUID/Ulid/整数）
- [ ] 配置用项目约定的方式（env/config struct）
- [ ] 依赖来自 workspace/monorepo 而非重复声明
- [ ] 符合项目的 lint/format 规则
- [ ] 符合项目的目录结构约定

---

## 维度审查顺序（按 ROI 参考，不跳过任何维度）

按 ROI 排序供审查顺序参考——时间紧时可先查前面的，但所有维度都必须审查，不存在可跳过的维度：

1. **AI 作弊模式**（见 ai-cheat-patterns.md）— 最高 ROI，5 分钟扫完
2. **正确性 & 安全**（第 1、5 维度）— 直接影响线上
3. **错误处理**（第 2 维度）— 决定可运维性
4. **测试有效性**（第 7 维度）— 决定未来可改动性
5. **可维护性 SOLID**（第 3 维度）— 决定长期成本
6. **性能**（第 6 维度）— 热路径优先（非热路径也必须解决，ROI 低于前 5 项）
7. **可读性/设计模式**（第 4、8 维度）— 长期价值，也必须解决

**不要纠结能自动化的事**：格式（prettier/rustfmt）、import 排序、lint（eslint/clippy）交给工具，审查聚焦设计/正确性/作弊/安全。
