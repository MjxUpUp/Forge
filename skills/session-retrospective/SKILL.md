# 会话经验沉淀

会话结束是经验最鲜活的时刻——上下文还在，踩过的坑、纠正过的判断都清楚。一旦压缩或新开会话，细节就靠 handoff 接力，必然丢。这个 skill 把"靠人临结束想起该记什么"变成"结构化回顾 + 载体路由"。

**核心原则：沉淀的目标是"未来不靠人记"。一条经验该进什么载体，取决于"能不能让机器替人记"。**

## When to Use / When NOT to Use

**Use when:**
- 一个主线任务/会话接近完成（功能做完、发布完、问题解决完）
- 用户说"回顾 / 沉淀 / 总结这次"
- 同一类坑这次又踩了（出现 ≥2 次的模式）
- 拿不准一条经验该进 memory 还是该做成 skill 还是该自动化

**SKIP:**
- 任务没结束（中途打断不沉淀，避免半成品经验）
- 纯 bug 修复且无普遍教训（直接 fix，不沉淀）
- 用户只要"行不行/对不对"的简单确认

## 流程（先读结论 + 5 步）

### 0. 先读结构化结论（forge act show）——锚定事实，别靠回忆

会话结束最容易丢的是"这次完成声明有多少实跑证据"。临结束凭记忆回顾会漏、会美化（"我觉得验证过了"）。先读 deterministic 结论再回顾：

- 跑 `forge act show`（最新任务结论）或 `forge act list`（全量，`--json` 给结构化）
- 看 **Strength / Ratio**：Unverified（零实跑）/Weak（声明主要靠 agent 自述）= 完成没真验证 → 这正是该优先回顾的盲区，分数看不出来
- 看 **LowDimensions**：<70 的评分维度直接是回顾靶点
- 结论带 RetrospectiveNudge 时已附一行 `→ session-retrospective:` 指令 → 按它回顾根因，再走下面载体决策树

这是 Act 反馈臂的入口：回顾从"agent 临结束想起什么"变成"读 forge 跑出来的结构化结论"。Strong 且 >=70 的干净完成不强制回顾证据（无盲区），但仍走下面 5 步看协作类教训。

### 1. 回顾——扫本次协作的非显然经验

逐项问：这次有什么是**开始时不知道、做完才明白**的？
- 踩过的坑 + 根因（不只是现象）
- 被纠正的判断（"我以为 A，实际 B"）
- 反复出现的模式（同类操作漏了第 N 次）
- 验证发现的意外（"本地过 ≠ CI 过""根 README 对但分发副本错"）

❌ 不沉淀：已知的常识、一次性无普遍性的细节、代码本身记录的东西（git log/结构）。

### 2. 提炼到可复用模式

把事件提炼成模式：
- ❌ "cursor 出现 37 个断链"（事件）
- ✅ "删 skill 源后必须核查**全部** agent 目标，不能只查本次改过的"（模式）

模式才有跨场景价值；事件只对此项目此时刻有用。

### 3. 载体路由（核心决策步骤）← 决策树见下

对每条经验，按决策树判定进哪个载体。**这是最关键的一步**——选错载体，经验要么靠不住（进 skill 但本该是代码），要么被淹没（进 memory 但本该是 skill）。

### 4. 落地

按路由写入对应载体：
- **memory**：一个文件一个事实，更新 `MEMORY.md` 索引。条目结构要让 AI 可执行（非人类叙述）：
  ```markdown
  ### <简短标题>
  **问题**: 什么场景触发 | **现象**: 出错症状 | **根因**: 为什么错
  **解决**: 对 AI 的指令（如"禁止在 Windows 用 bash sed"），不是对人类的描述 | **涉及**: 语言/框架
  ```
  写入位置按类型路由：语言/框架易错点 → 对应 skill 的 Gotchas；开发流程/协作规则 → AGENTS.md/CLAUDE.md 铁律段；跨项目通用 → Forge skills canonical；项目特定 → 项目 MD。
- **skill**：建 `skills/<name>/SKILL.md`，过 R1-R9 + 防注水扫描
- **代码/守卫测试**：写测试钉死一致性（转交 `docs-consistency-guard`）
- **hook**：注册到 `.claude/settings.local.json` + 脚本
- **CI 门禁**：加进 release/CI workflow 的前置 job

### 5. 验证载体生效

- memory：下次会话能否被召回（描述字段是否会被判相关）
- skill：`forge skills validate` 过 R1-R9、`audit` 过安全、触发描述是否会命中
- 代码/hook/CI：本地或 CI 实跑一次确认拦截/通过

## 载体决策树（核心）

按顺序问，命中即停：

| # | 问题 | 命中 → 载体 | 理由 |
|---|---|---|---|
| 1 | 能程序化、确定性检查吗？（如"文档数 == 注册数"） | **代码：守卫测试 / CI 门禁 / forge 命令** | 确定性比对靠人必漏，必须机器强制 |
| 2 | 需要判断、且跨项目可复用的流程吗？（如"如何排查根因""如何做发布验证"） | **skill** | 需判断无法完全程序化，靠 skill 引导 agent |
| 3 | 一次性、特定项目的事实/教训吗？（如"某 API 在这台机的坑"） | **memory** | 不跨项目、不需流程，记事实即可 |
| 4 | 每次工具调用都该查、高频低成本吗？（如"会话开始扫 skill 安全"） | **hook** | 高频必要，挂在工具事件上自动跑 |
| 5 | 发布/提交前才查、低频高价值吗？ | **CI 门禁** | 低频，只在发布/提交时拦 |

### 铁律：能自动化的不进 skill

skill 的成本是"靠 agent 遵循"——会漏。能用代码/CI/hook 机器强制的，**绝不**做成 skill。违反这条 = 产出"本该是代码的 skill"，既靠不住又占 description 触发空间和加载成本。

> 典型反例：把"发布前检查 README hook 数 == settings.go 注册数"做成 skill 提醒 → 错。它是确定性检查，该是守卫测试 + CI。skill 只剩"教人怎么写这个守卫测试"的方法论。

### skill 的额外门槛（建之前必过）

即便路由到 skill，也要先确认：
- **跨项目复用**？只此项目的 → 进 memory
- **需要判断**？完全可程序化的 → 进代码
- **高频/重要**？低频冷门的 → 不值得占 skill 空间
- **不与现有 skill 重叠**？重叠的 → 合并/补充，别新建

## Red Flags（出现这些想法 = STOP，你在选错载体）

- "这条记 memory 就行"——但它其实是可程序化的检查 → 该进守卫测试/CI
- "做个 skill 提醒一下"——但它是确定性检查 → 该进代码，skill 是错载体
- "经验太琐碎不值得沉淀"——但同类坑已踩 3 次 → 该进 memory 或 skill
- "以后记得查"——"记得" = 靠人 = 必然漏 → 该自动化
- "这条是 X 专属"——但核心方法论跨项目 → 通用部分进 skill，专属细节进 memory

## Rationalizations（堵借口）

| 借口 | 现实 |
|---|---|
| "memory 够用了" | memory 是背景召回，弱；反复踩的坑该升级成 skill 或代码 |
| "做成 skill 最灵活" | 灵活 = 靠人执行 = 会漏；能强制的用代码/CI/hook |
| "这条只此一次" | 踩过 2 次的"一次"就该沉淀；看模式不看个例 |
| "自动化太重" | 一次守卫测试 ~30 行防永久漂移，比"每次记得查"轻 |
| "以后再说" | 以后 = 永不；会话结束是沉淀成本最低的点 |
| "经验记下来就行不用验证" | 记下的载体若不生效（skill 不触发/memory 不召回）等于没记 |

## Gotchas（最高信号）

- **载体选错是最大浪费**：可程序化的进 skill = 产出"本该是代码的 skill"。先问"能自动化吗"，再问"进哪个载体"。
- **memory 的事实会过期**：memory 写的是"当时为真"——引用的文件/函数/数字（如"有 7 个 hook"）会随代码变。沉淀时若引用了具体标识，要当场校验仍成立；召回使用前也要复核。
- **skill 不是越多越好**：每个 skill 占 description 触发空间 + 加载成本。只在"跨项目 + 需判断 + 高频/重要"时建，重叠的合并。
- **提炼模式而非记事件**："cursor 断链"是事件，"删源后核查全部目标"是模式。沉淀模式。
- **会话结束是黄金点**：上下文鲜活；压缩/新开后靠 handoff 接力丢细节。任务一完成就回顾。
- **验证载体生效 ≠ 写完就完**：skill 要跑 validate/audit/防注水；memory 描述要能被相关性判定命中。不验证的沉淀等于没做。
- **跨 agent 目标的经验要全量核查**：删/改 canonical 源后，经验若涉及分发，要核查**所有** agent 目标（claude/cursor/pi），不只本次改的——会话内只改了 claude/pi，cursor 漏改是高频盲区。
- **经验要写成对 AI 的规则，非人类叙述**：写"我们遇到 XXX 问题"的叙述 = 无效沉淀；条目必须含"解决"字段且是对 AI 的指令。
- **跨项目通用经验别只写项目 MD**：只写项目级，换项目继续踩。通用经验写 Forge skills canonical 对应 skill 的 Gotchas，项目特定才写项目 MD。
- **写入位置错 = 加载不到**：领域知识写进 CLAUDE.md 但 CLAUDE.md 只在 init 时读 → 永不加载。领域知识写对应 skill，流程规则写 AGENTS.md/CLAUDE.md。

## 与其他 skill 的分工

- **docs-consistency-guard**：当路由判定"进守卫测试"时，转交它建具体的文档一致性测试。
- **session-continuity**：跨会话**恢复**（handoff 接力）。本 skill 是会话**结束**的沉淀。二者衔接：本 skill 产出的 memory/经验，是 session-continuity 下次恢复的素材。
- **skill-authoring-standard**：当路由判定"进 skill"时，用它写符合 R1-R9 规范的 skill。
- **code-review-gate**：审代码质量。本 skill 是经验沉淀，不审代码。
