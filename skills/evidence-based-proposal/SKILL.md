# 方案必须有依据

任何技术方案必须基于实际环境验证，不能基于通用行业知识直接推荐。

## 提方案前的两个必答题

在输出任何方案之前，先回答：

1. **实际环境是否支持这个方案的假设？**
   - 模型能力是否到位？（本地模型可能不支持工具调用、结构化输出）
   - API 是否存在？（crate 的 API 可能跟直觉不同）
   - 运行时行为是否符合预期？（网络、文件系统、进程管理）

2. **这个方案具体解决当前哪个断点？**
   - 不说"提升可维护性"这类空话
   - 指出具体的问题：第 X 行报错 Y、Z 功能缺少 W 能力

## 执行流程

```
遇到问题
  ↓
验证环境（读代码、跑命令、查文档）
  ↓
确认假设成立
  ↓
输出方案
```

**不是：**
```
遇到问题
  ↓
凭行业知识猜测
  ↓
输出方案
```

## 禁止的反模式

### ❌ 没看模型能力就出方案

实际推理引擎 `tool_calls` 永远返回 None，根本不支持工具调用，但方案基于工具调用能力设计。

**正确做法**: 先调一次 API 看 `tool_calls` 字段是否工作，再决定方案。

### ❌ 没查 crate API 就凭猜测写代码

`calamine::Reader::new()` 不存在，泛型参数写错，反复编译失败。

**正确做法**: 先 `cargo doc --no-deps -p <crate>` 或直接读源码，确认 API 签名。

### ❌ 基于通用知识推荐而不验证当前环境

"用 Hermes ChatML 格式"——实际模型根本不用这个格式。

**正确做法**: 查模型文档或测试实际输入输出，确认格式。

### ❌ 解决不存在的问题

"加一个缓存层提升性能"——但当前性能瓶颈不在这里。

**正确做法**: 先 profile 确认瓶颈在哪，再针对性优化。

## 检查清单

输出方案前：

- [ ] 方案依赖的能力/工具/API，在当前项目中是否存在？
- [ ] 方案解决的是用户描述的具体问题，还是我想象中的问题？
- [ ] 如果方案涉及新依赖，查过文档确认 API 用法了吗？
- [ ] 如果方案涉及本地模型，验证过模型实际支持这个能力吗？

## Common Rationalizations

| 借口 | 现实 |
|---|---|
| "行业都这么做" | 通用知识 ≠ 当前环境。模型/crate/API 可能与直觉不同，必须验证 |
| "这个 crate 应该有这个方法" | 应该 ≠ 有。`Reader::new()` 可能不存在、泛型参数可能不同，读源码/cargo doc 确认 |
| "先实现，遇到问题再调" | 问题到生产才暴露 = 昂贵。提方案前验证假设，成本低得多 |
| "这方案很标准" | 标准方案在错误环境下也会错。先回答"实际环境支持这个假设吗" |
| "性能问题用缓存解决" | 缓存解决的是"确认的"瓶颈。先 profile 定位瓶颈，别解决不存在的问题 |
| "用户说要做 X，我就设计 X" | 用户说的是 WHAT 不是 HOW。HOW 要基于环境验证，不是凭行业知识猜 |
| "API 文档我看过" | 看过 ≠ 当前版本有效。确认当前依赖版本下 API 签名 |

## Red Flags — STOP 先验证再提方案

如果你正要做这些，STOP：
- 凭"行业最佳实践"直接推荐库/框架/模式，没在当前项目验证
- 凭记忆写不熟悉 crate 的 API（先 cargo doc / 读源码）
- 假设本地模型支持某能力（工具调用/结构化输出），没调 API 验证
- 推荐优化方案但没 profile 确认瓶颈在哪
- 设计基于"应该"而非"实测"

**以上都意味着：你在猜不在验证。先跑命令/读代码/查文档拿证据，再提方案。**
