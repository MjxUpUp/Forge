# 方案脱离实际的案例

## 案例 1：没看模型就出 ChatML 方案

**场景**: 项目需要设计 prompt 格式与本地模型交互。
**做了什么**: 基于 Qwen 系列模型通常支持 Hermes ChatML 格式的行业知识，直接出了 ChatML 格式的 prompt 工程方案。
**实际**: 测试发现推理引擎（llama.cpp）的 `tool_calls` 字段永远返回 None，模型根本不支持工具调用。ChatML 格式的方案完全脱离现实。
**代价**: 浪费了一轮讨论时间，方案从零重新设计。
**教训**: 先调一次 API 验证能力，再设计格式。

## 案例 2：凭猜测使用 calamine crate

**场景**: 项目需要解析 Excel 文件。
**做了什么**: 不查文档直接写 `calamine::Reader::new()`，泛型参数凭直觉填。
**实际**: calamine 的 API 需要指定 Reader 类型（Xlsx、Xls 等），构造方式完全不同。
**代价**: 反复编译失败 3-4 轮。
**教训**: 使用新依赖前，先 `cargo doc --no-deps -p calamine` 或读源码。

## 案例 3：没有基准的优化方案

**场景**: 讨论项目性能优化。
**做了什么**: 提议加缓存层、加连接池等优化。
**实际**: 当前连基本功能都没跑通，性能瓶颈完全未知。
**教训**: 没有 profile 数据的优化方案是空中楼阁。

## 案例 4：用了不存在的类型变体

**场景**: 项目错误处理。
**做了什么**: 使用 `CoreError::ToolExecution`，编译失败。
**实际**: 实际变体名是 `ToolExecutionFailed { tool, message }`。
**代价**: 编译失败后才知道。
**教训**: 使用项目内自定义类型前，先 grep 确认变体名和字段名。
