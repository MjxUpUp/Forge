## 案例 1：补充调研在结论后追加独立章节

**场景**：用户要求补充腾讯技术工程的调研内容到已有的飞书文档（前六章已完成，含结论）。

**错误做法**：
- 在第六章"结论"之后追加了"七、国内厂商深度分析"
- 结论在前，新分析在后——逻辑顺序颠倒
- 用户评价："这个补充调研，实际上应该算作原本调研的一部分，你却在结论后面补充了章节，看起来是非常割裂的"

**正确做法**：
- 把腾讯调研的内容拆散：国内市场数据→融入第一章、国内产品→融入第二章、质疑验证→融入第四章、趋势补充→融入第五章
- 更新结论
- 用 overwrite 重写全篇

**对应流程**：research-workflow Phase 5（版本化补充融入）+ Phase 3 Gate（结构规划禁止结论后追加章节）。

**系统性解法**：此案例与案例 3 连修 4 轮的根因是"靠模型自觉遵守拆散融入规则"不可靠。Phase 5 已升级为**版本化改写**（见 references/versioned-rewrite.md）：fetch 原文快照 → 补充内容映射表 → 基于原文改写 → 发布前自检 Gate → overwrite 发布 → 版本归档。流程本身不允许 append，每步有产物，出错可回滚到 pre-merge 快照。

## 案例 2：格式不一致——没查就写

**场景**：前六章通过 overwrite 用 markdown 一次性写入飞书文档。第七章补充时用了 XML 富格式（彩色 callout、grid 分栏）。

**错误做法**：
- 没有先 fetch 检查已有文档的格式
- 直接用了不同的格式系统写入
- 用户评价："第七章的风格跟整个文档的怎么格格不入？"
- 修复后仍不满意："你不觉得你更新后文档排版奇怪吗？"

**正确做法**：
- 写入前先 `lark-cli docs +fetch --doc-format markdown` 查看现有格式
- 用相同格式写入

**对应流程**：research-workflow Phase 4（文档输出"先 fetch 查格式"）+ Phase 6 飞书发布。

## 案例 3：连续 4 次修复仍未解决根因

**场景**：同一个文档的格式/结构问题修了 4 轮。

**错误序列**：
1. 用 markdown append → 格式与前六章 XML 富格式不一致
2. 用 XML 重写 → 风格又与前六章 markdown 不一致（彩色 callout 满天飞）
3. 用 markdown 重写 → 格式统一了但结构割裂（独立第七章挂在结论后）
4. 融入前六章 → 终于正确

**根因**：每次修复只解决表面问题（格式），没理解用户反馈的真正含义（结构割裂）。

**预防**：在 Phase 3（结构规划）阶段就明确"补充内容融入现有章节"，避免进入"格式修复循环"。更根本的预防是走 Phase 5 版本化改写流程（见 references/versioned-rewrite.md），发布前自检 Gate 会拦住"结论后追加新章"。

**对应流程**：research-workflow 禁止反模式「连续修格式 ≠ 修结构」+ Phase 5 版本化改写自检 Gate。

## 案例 4：调研引擎未被触发

**场景**：用户说"帮忙调研一下，目前市面上已经出现了那么多种 agent 了"，模型手动逐个调 WebSearch 而不是走深度调研引擎。

**问题**：深度调研引擎（Phase 1）提供了 fan-out search + adversarial verify + cited report 的完整流程，但模型没有进入 Phase 1，退化成单次搜索堆砌。

**正确做法**：先判调研量级。
- 用户请求包含"调研""研究""分析"且需多源对比 + 出报告 → research-workflow Phase 1
- 用户只是查个事实/数据/对比/进展（需≥2源交叉但不出报告） → **fact-research**（轻量网络调研，别套用重量级流程）
- 开发中查 API/报错/库用法 → dev-lookup

**对应流程**：research-workflow Phase 0 模式判定 → Phase 1 引擎；轻量场景转 fact-research。

## 案例 5：凭记忆拼凑已有文档

**场景**：需要 overwrite 整篇文档时，模型没有先 fetch 获取当前完整内容，而是凭对话上下文中记忆的片段 + 新内容拼接。

**风险**：
- 已有内容可能被篡改（记忆中的措辞 ≠ 实际文档中的措辞）
- 可能遗漏已有段落
- 格式可能不一致（记忆中的格式判断可能有误）

**正确做法**：
1. 先 `lark-cli docs +fetch --doc-format markdown` 获取完整当前内容
2. 在获取到的实际内容基础上做修改
3. overwrite 回去

**对应流程**：research-workflow Phase 4（overwrite 前必须先 fetch 完整内容）。

## 案例 6：调研后没有自动发布飞书知识库

**场景**：调研结束时，只输出了本地md文档，没有进行飞书文档发布。

**风险**：
- 本地数据被删除
- 本地数据在线查询不到
- 存放位置不一致

**正确做法**：
1. 找到需要发布到的飞书目标位置，详细命令与异常处理见 **`references/lark-publish.md`**
2. 汇总调研总结作为本次发布的飞书主文档，多维度报告作为子文档

**对应流程**：research-workflow Phase 6 — 飞书发布。
