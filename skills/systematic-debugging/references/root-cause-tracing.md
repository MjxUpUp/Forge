# Root Cause Tracing — 向上回溯找源头

本文件是 systematic-debugging Phase 1（数据流追踪）的详细技术。

## 核心手法：反向追溯

错误深在调用栈里时，**从坏值出现处往回追，追到坏值的诞生源头**。在源头修，不在症状处修。

## 追踪步骤

1. **定位坏值出现处**——报错/异常的现场，那个错误的值是什么
2. **问：这个值从哪来？**——是参数传入？函数返回？全局状态？文件读的？
3. **追到上一级**——调用方传的什么？谁算出这个返回值？
4. **重复直到源头**——一直追到值的最初诞生处（用户输入/配置文件/外部 API/默认值）
5. **在源头修**——修诞生处，不修沿途传递

## 信号：你可能在修症状而非根因

- 修了一处，bug 在别处换形态出现
- 同样的值在多个地方错误 → 你在下游加校验而非追源头
- "我加个 null check 就好了" → 为啥是 null？null 从哪来？

## 多层系统的边界插桩模板

数据流经 N 层时，在**每层边界**打印进出数据，跑一次即可定位断裂层：

```
层 1 → 层 2 边界:
  - 打印传入层 2 的数据（完整值，不是 "已设置"）
  - 打印环境变量/配置（IDENTITY: ${X:+SET}${X:-UNSET}）
  - 记录层 2 入口状态

层 2 → 层 3 边界:
  - 同上

...跑一次，看哪层的数据变形/丢失/错误
```

**反模式**：凭直觉在某一层加修复，没数据证明是那层的问题。先收集证据定位层，再深入那层。

## 真实案例形状

- **"端口导致限流失效"**：限流 key 含 RemoteAddr:port，httptest 固定 RemoteAddr 测不出来 → 追到 key 生成处，发现生产环境每次请求端口不同导致 key 不命中
- **"配置没生效"**：env → workflow → build script → runtime 四层，逐层打印 env，发现 workflow→build 之间 env 没透传
- **"随机失败"**：坏值是竞态产生的共享状态 → 追到共享状态的读写处，发现无锁

## 与 compile-fix-loop 的区别

compile-fix-loop 修**编译期**错误（类型系统已告诉你根因，读 help/note 即可）。
本 skill 修**运行期**错误（编译过了但行为错，需要自己收集证据找根因）。

编译错误用 compile-fix-loop；行为 bug 用 systematic-debugging。
