# Systematic Debugging — 系统化调试

随机修复浪费时间、制造新 bug；快补丁掩盖根因。**永远先找根因再修复。** 修症状 = 失败。

**违反规则的字面就是违反规则的精神。**

## The Iron Law

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

没完成 Phase 1，不准提修复方案。

## When to Use

**任何技术问题都要用**：测试失败、生产 bug、意外行为、性能问题、构建失败、集成问题。

**尤其在这些时候不能跳过**：
- 时间压力下（紧急让人想猜）
- "就一个小修复"看着明显
- 已经试过多次修复
- 上一次修复没生效
- 自己没完全理解问题

**不要因为这些跳过**：
- 问题看似简单（简单 bug 也有根因）
- 赶时间（赶 = 保证返工）
- 想立刻修好（系统化比乱试更快）

## 四阶段（必须顺序完成）

### Phase 1: 根因调查（任何修复前）

1. **完整读报错**——不要只看第一行。报错常含确切解法，完整读 stack trace，记下行号/文件/错误码
2. **稳定复现**——能可靠触发吗？确切步骤？每次都发生？不能复现 → 多收集数据，不要猜
3. **检查最近变更**——`git diff`、近期 commit、新依赖、配置改动、环境差异
4. **多组件系统：在边界插桩收集证据**

   当系统有多个组件（CI→构建→签名，API→服务→数据库），**修复前在每个组件边界加诊断日志**：

   ```bash
   # 对每个组件边界：
   #   - 记录进入组件的数据
   #   - 记录离开组件的数据
   #   - 验证环境/配置传递
   #   - 检查每层状态
   # 跑一次收集证据 → 定位失败组件 → 深入调查该组件
   ```

5. **追踪数据流**（错误深在调用栈时）——见 `references/root-cause-tracing.md`。简版：坏值从哪产生？谁用坏值调用的？一直往上追到源头，在源头修而非症状

### Phase 2: 模式分析（修复前找模式）

1. **找工作的样例**——同代码库里类似的、能工作的代码。什么工作的和坏的相似？
2. **对照参考完整读**——实现某个 pattern 时，**完整读**参考实现，每行都读，不要略读
3. **识别差异**——工作的 vs 坏的，每个差异都列出，再小也别假设"这不可能有影响"
4. **理解依赖**——这还需要哪些组件？什么配置/环境？做了什么假设？

### Phase 3: 假设与测试（科学方法）

1. **单个假设**——清楚陈述："我认为 X 是根因，因为 Y"。写下来，要具体不要含糊
2. **最小化测试**——做**最小**改动验证假设，一次一个变量，不要同时修多个
3. **继续前验证**——生效？→ Phase 4。没生效？→ 新假设。**不要在上面叠加更多修复**
4. **不确定时**——说"我不懂 X"，不假装懂，求助，多调研

### Phase 4: 实现（修根因不修症状）

1. **创建失败测试**——最简复现，优先自动化测试。修之前**必须**有。写正经失败测试用 `tdd-cycle`
2. **单一修复**——只修 Phase 1 找到的根因。一次一改。不"顺手"改进。不捆绑重构
3. **验证修复**——测试现在过？没破坏其他测试？问题真解决了？
4. **修复没生效时**：STOP，数数试了几次
   - < 3 次：回 Phase 1，用新信息重新分析
   - **≥ 3 次：STOP 质疑架构（见下面 step 5）**，不要在没有架构讨论下尝试第 4 次修复
5. **3+ 次修复失败 = 架构问题**（不是假设失败）：
   - 信号：每次修复在不同地方暴露新的共享状态/耦合；修复需要"大规模重构"；每次修复在别处造新症状
   - STOP 质疑根本：这个 pattern 真的合理吗？是不是惯性硬撑？应该重构架构还是继续补症状？
   - **与人讨论后再试更多修复**

## Red Flags — STOP 回 Phase 1

如果你在想这些：
- "先打个快补丁，以后再查"
- "改 X 试试看好不好使"
- "一次多改几处，跑测试"
- "跳过测试，我手动验证"
- "大概是 X，我改一下"
- "我没完全懂但这个可能行"
- "pattern 说要 X 但我换个方式"
- "[没调查就]列主要问题+修复方案"
- 提方案前没追踪数据流
- **"再试一次修复"（已试 2+ 次）**
- **每次修复在不同地方暴露新问题**

**以上都意味着：STOP，回 Phase 1。**

**已 3+ 次修复失败 → 质疑架构（Phase 4.5）**

## 用户信号：你做错了

注意这些反馈：
- "这不是没发生吗？" → 你假设了却没验证
- "能给我们看吗？" → 你早该加证据收集
- "别猜了" → 你没理解就提修复
- "Ultrathink 这个" → 质疑根本，不只是症状
- "卡住了？"（沮丧）→ 你的方法没生效

**看到这些：STOP，回 Phase 1。**

## Common Rationalizations

| 借口 | 现实 |
|---|---|
| "问题简单，不需要流程" | 简单问题也有根因。流程对简单 bug 很快 |
| "紧急，没时间走流程" | 系统化比乱猜乱试**更快** |
| "先试这个，再调查" | 第一次修复定调子，从一开始就做对 |
| "确认修复生效后再写测试" | 没测试的修复不持久，先测试才有证明 |
| "一次多修几处省时间" | 无法隔离哪个生效，还造新 bug |
| "参考太长，我改用 pattern" | 一知半解必出 bug，完整读它 |
| "我看到问题了，修" | 看到症状 ≠ 理解根因 |
| "再试一次"（已失败 2+ 次）| 3+ 次失败 = 架构问题，质疑 pattern 别再修 |

## Quick Reference

| Phase | 关键活动 | 成功标准 |
|---|---|---|
| 1. 根因 | 读错误、复现、查变更、边界插桩 | 理解 WHAT 和 WHY |
| 2. 模式 | 找工作样例、对照 | 识别差异 |
| 3. 假设 | 立论、最小测试 | 确认或新假设 |
| 4. 实现 | 造测试、修、验证 | bug 解决、测试过 |

## 与其他 skill 的分工

- 纯**编译错误**（语法/类型/借用/生命周期）→ **compile-fix-loop**（本 skill 的编译特化分支）
- **测试失败**要先判断：是测试本身写错（弱断言）用 **test-discipline**；是代码逻辑 bug 用本 skill
- Phase 4 写失败测试时 → **tdd-cycle**（RED-GREEN-REFACTOR）
- 修复后声称完成前 → 验证用 **verification-before-completion** 理念（evidence before claims）
- 多 agent 调试复杂问题 → **agent-delegation**（两阶段审查）
