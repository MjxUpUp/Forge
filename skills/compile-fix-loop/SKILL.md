# 编译报错修复闭环

打破"写代码→编译→报错→粘贴→修复→新报错"循环的方法论。

## 核心原则

**你不是传话筒。你是编译器到 AI 之间的那层智能。**

用户粘贴报错的本质是：AI 不主动编译 → 用户被迫当人工 IPC 中继。

## 编译门禁前置（防患于未然）

**提交/声称完成前，主动跑全量编译检查**，不要等用户粘贴报错。各语言默认命令：

| 语言 | 全量编译命令 | 说明 |
|---|---|---|
| Rust | `cargo check --workspace --all-targets` | 比 build 快，只检查不生成产物 |
| Go | `go build ./...` + `go vet ./...` | build 报编译错，vet 报静态可疑 |
| TypeScript | `tsc --noEmit` 或 `npx tsc --noEmit -p .` | noEmit 只类型检查不产出 |
| ArkTS | DevEco `hvigorw assembleHap` 或 IDE 构建 | 鸿蒙专用 |

**关键**：`go build`/`cargo check` 一次只报少量错（编译器在第一个致命错处停下）。修复后必须**重跑**直到零错误——不要修一个就以为完了。这是"修一个错引下一个错"循环的根因。

## 修复流程

### Step 1: 理解根因，不盲目修改

拿到编译报错后：
- 读完整报错信息（不要只看第一行）
- 理解报错类型：语法错误 / 类型错误 / 借用检查 / 生命周期 / 链接错误 / 依赖冲突 / **未定义符号** / **模块缺失** / **rune 字面量**
- 定位到具体文件和行号
- **在脑中模拟为什么这个代码会导致这个错误**，不要直接改

### Step 2: 检查是否同类错误

搜索项目中有没有相同模式的代码也会触发同类错误：
- 同一文件的其他地方
- 其他文件的类似调用
- 本次会话中之前是否出现过相同类型的错误

### Step 3: 修复 + 验证

- 一次性修复所有同类错误，不要逐个修改
- 修改后**主动运行编译检查**，不要等用户粘贴下一轮报错
- 如果是跨平台问题（Windows vs Linux），确认当前运行环境后再修改

### Step 4: 如果修复后出现新错误

判断新错误类型：
- **类型 A：我的修复引入了新错误** → 我的修复方式有问题，回到 Step 1
- **类型 B：修复暴露了原本被掩盖的其他错误** → 这些错误本来就存在，继续修复
- **类型 C：全新错误，与修复无关** → 可能是环境变化，检查上下文

### Step 5: 蒸馏经验

同一类错误出现 ≥2 次时：
- 提取到对应 skill 的 Gotchas
- 如果是平台相关问题，补充到项目 AGENTS.md 或 memory
- 如果涉及第三方库 API 用法，记录到 AGENTS.md

## Gotchas

### 问题: 改了一个地方但没改同类错误
**现象**: 修复了一处编译错误，编译通过，但同类代码在其他文件里还是旧的写法
**解决**: 修复前用 `grep` 搜索相同模式，一次性修完所有同类错误

### 问题: 不看完整报错信息
**现象**: 只读报错的第一行，修改后报出更多错误
**解决**: 完整的编译错误输出通常包含根因分析（如 Rust 的 `help:` 和 `note:`），必须全部读完

### 问题: 路径分隔符在不同平台表现不同
**现象**: Windows 上用 `/`、Linux 上用 `\` 导致编译失败
**解决**: 涉及路径操作时，使用语言标准库的路径 API（`std::path::Path`、`path.join()`），不要手写 `/` 或 `\`

### 问题: 修复了编译错误，但测试失败
**现象**: 改了代码让编译通过，但把逻辑改错了
**解决**: 编译通过 ≠ 正确。修复后立即运行相关测试。如果修复涉及逻辑变更，先确认逻辑正确性再编译

### 问题: Go 编译"修一个错引下一个错"循环
**现象**: 修了 `undefined: foo`，重编译又报 `undefined: bar`，循环 5+ 轮
**根因**: Go 编译器遇致命错即停，一次只露一个；且 Go 的 `undefined:` 常因 import 缺失（一个 import 漏了 → 该文件所有引用全 undefined）
**解决**: ① 看到 `undefined: X` 先查 X 是不是漏 import（`import "strings"` 漏了 → 所有 `strings.X` 报 undefined），一次补全 ② 用 `go vet ./...` 补静态分析 ③ 修完重跑 `go build ./...` 直到零错误，不要修一个就停

### 问题: 同一会话反复粘贴同类编译报错
**现象**: 用户反复粘贴 Go/Rust 编译报错，每次修一个又来一个
**根因**: 没执行"编译门禁前置"——声称完成前没主动跑全量编译
**解决**: 每次改完代码，主动跑 `cargo check --workspace` / `go build ./...` / `tsc --noEmit`，把所有错误一次收齐批量修，不要等用户中继
