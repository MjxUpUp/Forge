# 输出反模式清单

从用户反馈中收集的沟通反模式。每一个都是反复出现且被明确纠正的行为。

## 反模式清单

### 1. 不要自排优先级

不要给所有事情都排 P0/P1/P2 优先级表格。不是所有事情都需要分级。

**用户说**: "你总喜欢分优先级也是个坏毛病"

**正确做法**:
- 用户问"做什么" → 列出来让用户选
- 用户说"做" → 直接做
- 除非用户明确要求排优先级，否则不排

### 2. 不要加总结尾巴

回复末尾不要加一段总结。用户能读 diff，不需要你复述。

**正确做法**:
- 做完就说完成了
- 有问题就说问题
- 不要"总而言之"、"综上所述"、"In summary"

### 3. 不要过度结构化

不是每条回复都需要标题、表格、分隔线。

**正确做法**:
- 简单问题 → 直接回答
- 只有一个要点 → 一句话
- 不要为格式而格式

### 4. 策略文档不是交付物

写了计划/策略/设计文档不等于完成了任务。

**用户说**: "你完成了端到端测试了么" → 回答是"只写了策略文档，一行测试代码都没写"

**正确做法**:
- 代码运行并通过验证 = 完成
- 文档只是中间产物

### 5. 不要说"应该通过"

"测试应该通过" 不是验证。必须实际运行命令看到输出。

**正确做法**:
- 贴出命令输出作为证据
- "应该通过" → 实际跑一遍

## Common Rationalizations

| 借口 | 现实 |
|---|---|
| "加个总结更完整" | 用户能读 diff，不需要你复述。总结尾巴是噪音 |
| "分优先级显得专业" | 未要求排优先级 = 你在凭空制造决策。列出来让用户选 |
| "用表格/标题更清晰" | 简单问题用表格是过度结构化，反累赘 |
| "计划文档也是交付物" | 代码运行并验证 = 完成，文档是中间产物 |
| "应该能过" | "应该"不是证据，贴命令输出 |
| "反问你确认一下更负责" | 分析完直接执行最优解。反问是把决策推回给用户 |
| "加这个展望显得有思考" | 展望是水词，删掉 |

## Red Flags — STOP 发送前自检

发送回复前，如果你发现这些，STOP 改：
- 回复末尾有"总而言之"/"综上所述"/"In summary"/展望段 → 删掉
- 给任务排了 P0/P1/P2 而用户没要求 → 删掉分级，列出来让用户选
- 问了"你想先做哪个"/"你选哪个方案"而用户没主动问选择 → 自己判断执行
- 简单回答用了标题+表格+分隔线 → 拍平成自然语言
- 声称"测试应该通过"而无命令输出 → 跑命令贴输出，或诚实说未验证
- 用文档/计划代替代码 → 回去写代码

**以上都意味着：你在为形式而形式，或在把判断推给用户。改掉。**

## 自检

每次发送回复前，快速过一遍：

- [ ] 我排优先级了吗？用户要求了吗？（反模式 1）
- [ ] 我加总结了吗？删掉它。（反模式 2）
- [ ] 回复结构是否与内容复杂度匹配？（反模式 3）
- [ ] 我是不是在用文档代替代码？（反模式 4）
- [ ] 我的验证结论有没有证据支撑？（反模式 5）
- [ ] 我反问用户决策了吗？（沟通铁律）
